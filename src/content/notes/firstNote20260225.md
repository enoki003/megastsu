# 1.概要
ポートフォリオ兼研究ノートサイトを作成した。
本サイトでは、コンテンツ駆動型の静的サイトジェネレーターであるAstroを採用し、低レイテンシかつ保守性の高い構成を実現した。
また、デプロイの効率化のため、レンタルサーバーへの自動デプロイを行うCI/CDパイプラインをGitHub Actionsで構築した。これにより、長期的に記事を投稿可能な運用環境と、高い拡張性および保守性を確保した。
本記事では、その背景から設計、課題および反省までを記述する。

# 2.導入
研究活動や学習記録を継続的に蓄積・公開するにあたり、既存のSNSや外部ブログサービスでは、長期的な整理性や設計自由度に制約があると判断した。
私の関心領域は分野横断的であり、単発の発表や一時的な登壇では継続的な文脈共有が困難である。
また、研究者・企業関係者・共同開発者に対し、自身の取り組みや技術的背景を一貫した形で提示する必要があった。
以上を踏まえ、自己紹介・研究ノート・ポートフォリオを統合した独自サイトの設計および開発を行った。

# 3.要件定義
## 3.1.目的
本サイトは、私の研究活動及び学習記録を長期的に蓄積・公開すると同時に、自己紹介およびポートフォリオとして機能させることを目的とする。

## 3.2.機能要件
本サイトが満たすべき機能要件は以下のとおりである。
### 3.2.1.記事管理機能
- Markdown形式またはその他の形式で記事を作成できること
- 記事の公開/非公開を切り替えられること
- 記事本体にメタデータを付与できること

### 3.2.2.コンテンツ分離機能
- タグやカテゴリにより分類が可能であること
- Note（学術寄り）とJournal（ブログ寄り）を分離して個別に確立すること

### 3.2.3.更新機能
- 記事作成から公開までに要するステップ数を少なくすること
- 書いている途中の記事を定期的に保存できるようにすること
- 記事の整理がしやすいようにすること

## 3.3.非機能要件
本サイトの非機能要件は以下である。
### 3.3.1.性能要件
- クライアント側に余計な負荷をかけないこと（JavaScriptの使用を最小限に抑えること）
- 高速に表示できること
- 低帯域環境でも表示できるようにすること

### 3.3.2.保守要件
- コンテンツとレイアウトを分離できるようにすること
- 機能を拡張しやすい設計にすること

### 3.3.3.運用コスト
- デプロイ作業を自動化すること

### 3.3.4.セキュリティ
- 不要な通信を避けること
- SSH鍵が漏洩しないようにすること

## 3.4.制約
本サイトには以下の制約が存在する。
- ホスティング環境はさくらのレンタルサーバである。
- 開発から運用、保守はすべて一人である。
- 大規模なトラフィックを想定しない
- 予算はサーバ代のみとする。
- 開発期間を短くすること

# 4.技術と手法
## 4.1.フレームワークの選定理由
今回は、コンテンツ駆動型のWebサイトジェネレーターである[Astro](https://astro.build/)を採用した。理由として、雛形が充実しており、学習コストを抑えて迅速に作成ができること、クライアントにJavaScriptなどの処理による負荷をかけないこと、Markdownやその他のソースを取り込める基盤が整っていることが挙げられる。また、あくまで記事がメインであり、複雑な実装を要さない。

## 4.2.CI/CDパイプライン
### 4.2.1.選定理由
一人で記事の投稿と保守をしていくので、CI/CD（継続的インテグレーション／継続的デリバリー）パイプラインの実装が必要であると考えた。
そこで、今回はGitHub上で管理できることから、[GitHub Actions](https://github.co.jp/features/actions)での実装を選定した。

### 4.2.2.デプロイメントのフロー
本サイトのCI/CDはGitHub Actionsにより自動化している。ワークフローは `push` および `pull_request` をトリガーとして起動するが、本番環境へのデプロイは `master` ブランチへの `push`（= PRのmerge後）でのみ実行される。

**トリガーと実行範囲**
- pull_request（対象: master）: ビルドと検証のみ（デプロイは実行しない）
- push（対象: master）: ビルドと検証に加え、本番デプロイまで実行する

**CIフェーズ（PR / push 共通）**
1. リポジトリをcheckoutする（actions/checkout）。
2. Node.js 20をセットアップし、npmキャッシュを有効化する（actions/setup-node）。
3. 依存関係をクリーンインストールする（npm ci）。
4. Astroのビルドを実行し、成果物を `dist/` に生成する（npm run build）。
5. 成果物の最低条件を検証する（`dist/index.html` の存在チェック）。

**CDフェーズ（push to master のみ）**

6. known_hosts をSecretsから配置し、SSH接続先のなりすましを防止する（StrictHostKeyChecking有効）。
7. ssh-agentを起動し、Secretsで管理するデプロイ用秘密鍵をロードする。
8. デプロイ対象ディレクトリにマーカー（`.deploy-root`）が存在することを検証し、誤配備を防ぐ。
9. rsyncで `dist/` をデプロイ先へ同期し、不要になった旧ファイルは `--delete` で削除する（マーカーは除外）。

この構成により、PR段階では安全にビルド検証のみを行い、master反映時のみ本番更新することで、運用時のエラーを低減できる。


## 4.3.記事獲得機能
本サイトの Note 及び Journal で使われている、特定のディレクトリからMDファイルを獲得し、記事一覧として表示する機能について解説する。

### 4.3.1. Contents Collections
 Astro には [Content Collections](https://docs.astro.build/ja/guides/content-collections/) という専用の機能が用意されている。

### 4.3.2.選定理由
本サイトは個人で開発・運用するため、外部CMSの導入や独自のMDパーサ実装はコストが高い。
また、単純にMDファイルを読み込むだけでは、frontmatterの形式や必須項目の保証ができず、長期運用において整合性が崩れるリスクがある。
AstroのContent Collectionsでは、frontmatterに対してスキーマを定義し、型レベルで検証を行うことができる。
これにより、形式不備や必須項目の欠落をbuild時に検出でき、コンテンツの完全性を担保できる。
さらに、コンテンツを構造化データとして扱えるため、一覧生成やフィルタリング処理も簡素化される。
以上の理由から、Content Collectionsを採用した。

# 5.結果
## 5.1.機能要件の達成状況

本サイトは、3章で定義した機能要件をすべて満たしている。以下に各要件と実装の対応関係を示す。

### 記事管理機能
Markdown形式の記事作成については、AstroのContent Collectionsを用いることで実現した。
各記事は `src/content` 以下に配置され、frontmatterによってタイトル・日付・公開状態などのメタデータを付与している。
また、公開/非公開の切り替えは `draft` フラグによって制御しており、ビルド時に公開対象の記事のみを抽出する構成とした。

### コンテンツ分離機能
NoteおよびJournalの分離については、ディレクトリ単位でコンテンツを分割することで実現した。
それぞれ独立したコレクションとして管理することで、一覧生成やルーティング処理を明確に分離している。
また、タグやカテゴリについてはfrontmatterで定義し、フィルタリング可能な構造とした。

### 更新機能
記事作成から公開までのフローは、ローカルでMarkdownを編集し、GitHubへpushするのみとした。
これにより、公開までのステップ数を最小化している。
また、Gitによるバージョン管理により、編集中の記事の保存および履歴管理が可能である。
さらに、ディレクトリ構造とメタデータによって記事整理が容易な構成となっている。

## 5.2.非機能要件の評価

本サイトは、非機能要件においても設計意図に沿った特性を実現している。

まず性能面では、Astroによる静的サイト生成を採用することで、クライアント側でのJavaScript実行をほとんど必要としない構成とした。
そのため、初期表示はHTMLと最小限のCSSのみで完結し、低帯域環境においても高速に表示可能である。

また、ビルド処理はCI環境上で自動実行され、依存関係のインストールからビルド完了まで一貫した再現性を持つ。
これにより、環境差異による不具合を排除している。

保守性の観点では、コンテンツ（Markdown）と表示ロジック（Astroコンポーネント）を明確に分離しており、機能追加やデザイン変更時の影響範囲を限定できる構造となっている。


## 5.3.開発・運用効率

CI/CDパイプラインの導入により、開発および運用フローは大幅に簡略化された。

記事の公開は、ローカルでMarkdownを編集しGitHubへpushするのみで完結する構成とした。
GitHub Actionsが自動的にビルドおよびデプロイを実行するため、手動でのサーバ操作は不要である。

また、pull request時にビルド検証を実施することで、構文エラーやビルド失敗を事前に検知できる。
これにより、本番環境への不完全なデプロイを防止している。

さらに、rsyncによる差分デプロイと不要ファイル削除（--delete）により、
デプロイ先の状態を常にリポジトリと一致させることができ、運用時の不整合を防ぐ構成とした。


## 5.4.制約条件下での実現性
さくらのレンタルサーバ環境上で問題なく動作しており、
特別なサーバサイド実行環境を必要としない静的構成とすることで、環境依存性を排除した。

また、外部CMSや有料サービスを使用せず、既存のGitHubおよびレンタルサーバのみで構成しているため、運用コストはサーバ料金のみに限定されている。

さらに、設計および実装はすべて個人で完結しており、
複雑な依存関係や高度な運用手順を排除することで、長期的に持続可能な構成となっている。

開発期間についても、Astroの既存機能およびGitHub Actionsを活用することで、
短期間での初期リリースを実現した。

# 6.今後の課題
## 6.1.記事一覧機能における課題
現状の記事一覧機能は最小限の構成にとどまっており、ページングやタグによる絞り込み機能を備えていない。
記事数が増加した場合、利用者が目的の記事へ到達しにくくなり、サイト全体の探索性および再利用性が低下するおそれがある。
そのため、今後は記事数の増加を前提とした一覧設計へ見直しを行う必要がある。

## 6.2.学術的記述における表現上の課題
本サイトの記事は学術的な内容を含むため、引用文献や注釈、構造化された記述を十分に表現するには、現在採用しているMarkdownベースの運用には限界がある。
とくに、今後より厳密な記述や複雑な文書構造を扱う場合には、記法と表示基盤の両面で拡張が必要になる。
この点は、今後のコンテンツ運用を考えるうえでの重要な課題である。

# 7.まとめ
本記事では、ポートフォリオ兼研究ノートサイトの設計および開発について述べた。Astroを用いた静的サイト構成と、GitHub ActionsによるCI/CDパイプラインを組み合わせることで、低コストかつ保守性の高い運用基盤を構築した。
特に、Content Collectionsによるコンテンツ管理と、pushベースで完結する公開フローにより、個人開発環境においても継続的な記事運用が可能な構成を実現した。
また、静的生成を前提とすることで、クライアント負荷の低減にも寄与している。
一方で、記事数の増加に伴う一覧性・探索性の低下や、学術的記述における表現力の不足といった課題も明らかになった。
以上より、本システムは個人による長期的な知識蓄積および公開を目的としたWeb基盤として有効であるが、運用規模の拡大および記述内容の高度化に対応するためには、継続的な改善が必要である。
